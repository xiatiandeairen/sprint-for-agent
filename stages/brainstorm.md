# Brainstorm

核心价值：**对齐**。用户的需求是模糊的 — 想法模糊、语言模糊、边界模糊。brainstorm 逐层消除偏差，直到双方对「要做什么」的理解完全一致。

> 本文件是执行指令。必须按「步骤」段逐步执行，每步验证「完成标志」后才进下一步。

<HARD-RULE>
brainstorm 的价值在对话过程，不在产出文档。
对齐的标准：用一句结论 + 一个具体例子描述需求，用户说「对，就是这个意思」。
如果做不到这个标准，说明还没对齐。
</HARD-RULE>

## 输入

| 字段 | 来源 | 必需 | 校验 |
|------|------|------|------|
| 任务描述 | 用户输入 | 是 | 非空 |
| sprint ID | entry.md | 是 | DB 中存在 |

## 输出

| 产出 | 路径 | 消费方 | 聚焦内容 |
|------|------|--------|---------|
| 约束 | `{sprint}/anchors/brainstorm.md` | 执行器 anchor-check | 边界/排除/标准/假设 |
| 交接 | `{sprint}/handoffs/brainstorm.md` | research / design / plan | 需求点+验收标准 |
| 摘要 | `{sprint}/reports/brainstorm.md` | 人 | 结论+例子+scope |

## 对齐层次

```
Layer 1: 想法对齐 — 用户脑子里想的和嘴上说的不一样
Layer 2: 问题对齐 — 表面需求背后的真实问题是什么
Layer 3: 目标对齐 — 做完之后什么变了，怎么衡量
Layer 4: 范围对齐 — 做什么、不做什么、边界在哪
Layer 5: 具象对齐 — 用具体例子验证理解一致
```

## 步骤

### Step 1: 读上下文

**做什么**: 了解现状，为对话提供素材
**怎么做**: 读相关代码/文档/近期 commit
**完成标志**: 能列出 >= 3 个现状事实（文件路径/接口名/数据流）

### Step 2: 想法对齐 [STOP:respond]

**做什么**: 把用户模糊的想法变成清晰的表述
**怎么做**:
一次只问一个问题。用自己的话复述用户的需求，问「我理解对吗？」

- 如果用户说「不完全对」→ 追问哪里不对，修正理解
- 如果用户说「对」但表述很短/模糊 → 追问细节：「你说的 X 具体是指什么？」
- 如果用户给了很长的描述 → 帮用户提炼：「你说了 A/B/C 三件事，核心是哪个？」

**完成标志**: 能用 2-3 句话复述需求，用户确认「对」。

### Step 3: 问题对齐 [STOP:respond]

**做什么**: 从表面需求追溯到真实问题
**怎么做**:
- 「为什么要做这个？」
- 「{回答} 导致了什么实际问题？」
- 「如果不做这个，会发生什么？」
- 追问到能量化影响：频率、耗时、影响范围

**完成标志**: 能区分「用户要做的事」和「要解决的问题」。两者可能相同（需求明确时），也可能不同（需求模糊时）。无论哪种，都记录下来。

### Step 4: 目标对齐 [STOP:respond]

**做什么**: 确定做完之后的验收标准
**怎么做**:
- 「做完之后，你怎么知道做对了？」
- 「有没有一个具体场景，做完后体验会不一样？」
- 「如果只能改善一件事，是什么？」

**完成标志**: 至少 1 个可验证的成功标准（「执行 X 命令看到 Y 结果」或「用户能做到 Z」）。不接受「更好」「更方便」等模糊目标。

### Step 5: 范围对齐 [STOP:respond]

**做什么**: 确认做什么、不做什么
**怎么做**:
- 「除了你说的 X，还有没有相关的要一起做的？」
- 「有没有你觉得应该做但其实可以不做的？」
- 「有没有绝对不能碰的？」

对用户的每个回答，判断是否改变了 scope：
- 新增了 → 记录
- 去掉了 → 记录
- 不变 → 明确确认

**完成标志**: scope 清单（做什么 + 不做什么），且至少有一个排除项。

### Step 6: 具象对齐 [STOP:confirm]

**做什么**: 用一句结论 + 一个具体例子验证对齐
**怎么做**:
用以下格式呈现给用户：

```
结论: {一句话描述核心需求}

例子: {一个具体的使用场景}
  当前: {现在是什么样}
  之后: {做完后是什么样}
  验证: {怎么确认做对了}
```

问用户：「这个描述准确吗？有没有哪里不对？」

**完成标志**: 用户对结论 + 例子都确认「对」。如果用户说「不完全对」，修正后重新呈现，直到确认。

### Step 7: 写三份产出

**做什么**: 按消费方写三个文件
**怎么做**:

**1. 约束文件** `{sprint}/anchors/brainstorm.md` — 给执行器 anchor-check 用：

```markdown
# Brainstorm Anchor -- {主题}

## 成功标准
- {可验证的标准}

## Scope
做: {列出}
不做: {列出}

## 假设
- [ ] {待验证}
```

**2. 交接文件** `{sprint}/handoffs/brainstorm.md` — 给 research/design/plan 消费：

```markdown
# 需求 -- {主题}

## 核心问题
{一句话}

## 具体需求

### 需求 1: {标题}
描述: {做什么}
验收: {怎么验证}

### 需求 N: ...

## Scope
做: {列出}
不做: {列出}

## 假设
- {待验证}
```

**3. 摘要文件** `{sprint}/reports/brainstorm.md` — 给人看：

```markdown
# Brainstorm -- {主题}

> 状态: [ok] 已确认

## 结论
{Step 6 的一句话}

## 例子
当前: {现在是什么样}
之后: {做完后是什么样}

## 背景
原始描述: {用户最初说的}
核心问题: {对齐后的问题}
影响: {量化数字}

## Scope 调整
{做什么 / 不做什么 / 调整了什么}
```

**完成标志**: 三个文件都存在

### Step 8: 用户确认 [BRANCH]

**做什么**: 请用户 review 摘要文件

| 条件 | 路径 |
|------|------|
| auto 模式 | → [INFO] 输出摘要要点，继续 |
| step-by-step 模式 | → [STOP:confirm] 呈现 `{sprint}/reports/brainstorm.md`，等用户确认 |

**完成标志**: auto: 已输出 / step-by-step: 用户确认 `[ok]`，摘要文件状态行更新

## 提前终止

| 条件 | 行为 |
|------|------|
| 用户说「不需要 brainstorm」 | skip |
| Step 2 中用户需求已经非常具体清晰 | 压缩 Step 3-5 为快速确认，不跳过 Step 6 |

## 验证清单 [CHECKLIST]

- [ ] `{sprint}/anchors/brainstorm.md` 存在 + 成功标准非空 + 排除项非空
- [ ] `{sprint}/handoffs/brainstorm.md` 存在 + 每个需求点有验收方式
- [ ] `{sprint}/reports/brainstorm.md` 存在 + 状态行 `[ok]`
- [ ] reports 中有结论 + 例子
- [ ] [人工] 用户确认「对，就是这个意思」

## 排查指南

**1. 用户说「就按我说的做」**
症状: Step 2-3 被跳过
处理: 尊重用户，但 Step 6 不能跳 — 即使用户需求明确，也要用结论+例子验证一次

**2. 对齐后 scope 失控**
症状: Step 5 后需求变成原来 3 倍
处理: 「这些都有价值，但建议分成多个 sprint。这次先做哪个？」

**3. 用户给不出成功标准**
症状: Step 4 反复说「就是好用就行」
处理: 给用户选项 — 「是 A 这样算好用，还是 B 这样？」用具体例子代替抽象问题
