# Design

核心价值：**具象化** — 在动手之前把方案变成看得见的设计。阻止 AI 边写边想、后期重构。

> 本文件是执行指令。必须按「步骤」段逐步执行，每步验证「完成标志」后才进下一步。

<HARD-RULE>
design 的价值在于 plan 阶段能直接基于 design 产出拆 chunk，不需要补充任何设计信息。
如果 plan 阶段还要做设计决策，说明 design 没做到位。
</HARD-RULE>

## 输入

| 字段 | 来源 | 必需 | 校验 |
|------|------|------|------|
| research handoff | research 阶段 | 有则读 | `{sprint}/handoffs/research.md` 存在 |
| brainstorm handoff | brainstorm 阶段 | 有则读 | `{sprint}/handoffs/brainstorm.md` 存在 |
| 任务描述 | 用户 / entry | 是 | 非空 |
| sprint ID | entry.md | 是 | DB 中存在 |

## 输出

| 产出 | 路径 | 消费方 | 聚焦内容 |
|------|------|--------|---------|
| 约束 | `{sprint}/anchors/design.md` | plan/execute 对照 | 接口契约 + 依赖约束 + 不碰列表 |
| 交接 | `{sprint}/handoffs/design.md` | plan 消费 | 按类型的具体设计 |
| 摘要 | `{sprint}/reports/design.md` | 人确认 | 设计概要 + 关键图表 |

## 设计类型识别

进入时根据任务内容识别类型（可多选）：

| 类型 | 触发信号 | 必须产出 |
|------|---------|---------|
| **UI/交互** | 涉及界面、用户操作流 | UI 示意图 + 交互效果说明（状态变化、操作步骤、反馈） |
| **架构/代码** | 涉及模块、接口、数据流 | 文件结构表 + 接口定义 + 按需：架构图/时序图/流程图/依赖图 |
| **策略/算法** | 涉及评分、排序、匹配、决策逻辑 | 算法描述 + 输入输出定义 + 边界 case + 具体参数/阈值 |
| **协议/规范** | 涉及数据格式、API 契约、命名约定 | 规范条目 + 示例 + 违反检查方式 |

## 步骤

### Step 1: 读上下文 + 识别类型

**做什么**: 读上游产出，确定设计类型
**怎么做**: 读 research handoff 的方案决策 + 相关代码
**完成标志**: 能说出「这次设计是 X 类型（+ Y 类型）」

### Step 2: 按类型产出设计

**做什么**: 针对每个识别到的类型，产出对应的具体设计
**怎么做**:

**UI/交互类**:
- 描述每个界面/状态的布局
- 交互流程：用户操作 -> 系统响应 -> 状态变化
- 边界状态：空状态、错误状态、极端数据

**架构/代码类**:
- 文件结构表（新建 / 修改 / 不碰）
- 接口定义（public API 签名，改已有接口标注 before/after）
- 依赖方向（模块间依赖 + 禁止方向）
- 按需画图（架构图/时序图/流程图/依赖图 — 用文字描述即可，不需要图片）

**策略/算法类**:
- 算法步骤（伪代码级别）
- 输入输出定义（类型 + 范围）
- 边界 case（空输入、极端值、并发）
- 具体参数/阈值及选择理由

**协议/规范类**:
- 规范条目列表
- 每条有示例（正确 + 错误）
- 违反检查方式（命令或正则）

**完成标志**: 每个类型的必须产出都有内容（不是占位符）

### Step 3: Self-review

**做什么**: 检查设计能否直接驱动 plan
**怎么做**:
- 架构类：文件结构表是否完整（能否直接拆 chunk）？接口签名是否清晰？
- UI 类：交互流程是否覆盖所有用户操作？
- 策略类：算法是否有具体参数（不是「合适的阈值」）？
- 协议类：规范条目是否可检查？
- 通用：不碰列表是否覆盖上游排除项？

**完成标志**: 以上检查通过，或缺项已补充

### Step 4: 写三份产出

**做什么**: 按消费方拆分

**约束文件** `{sprint}/anchors/design.md`:
```markdown
# Design Anchor

## 接口契约
{public API 签名列表，改已有接口标注 before/after}

## 依赖约束
{允许的依赖方向 + 禁止的依赖方向}

## 不碰列表
{不触碰的文件/目录}

## 类型特定约束
{UI: 交互规范 / 策略: 算法参数约束 / 协议: 规范条目}
```

**交接文件** `{sprint}/handoffs/design.md`:
```markdown
# Design -- {主题}

## 设计类型: {UI/架构/策略/协议}

{按类型的完整设计内容}

## 文件结构
| 操作 | 文件 | 职责 |
|------|------|------|

## 结构性约束
| 约束 | 检查命令 | 预期 |
|------|---------|------|
```

**摘要文件** `{sprint}/reports/design.md`:
```markdown
# Design -- {主题}
> 状态: [ok]

## 设计概要
{一段话概括设计方案}

## 关键决策
{列出设计中做的关键选择 + 理由}

## 影响范围
{涉及的模块/文件列表}
```

**完成标志**: 三个文件存在

### Step 5: 用户确认 [BRANCH]

**做什么**: 请用户 review 设计

| 条件 | 路径 |
|------|------|
| auto 模式 | → [INFO] 输出设计要点摘要，继续 |
| step-by-step 模式 | → [STOP:confirm] 呈现 reports/design.md + handoff 中的关键图表/接口，等用户确认 `[ok]` |

**完成标志**: auto: 已输出 / step-by-step: 用户确认 `[ok]`

## 提前终止

| 条件 | 行为 |
|------|------|
| 用户说「不需要设计」 | skip -> 进 plan |
| research 决策已包含足够设计细节 | 简化 -> 只补充缺失部分 |

## 验证清单 [CHECKLIST]

- [ ] 三份文件存在 + reports 状态 `[ok]`
- [ ] 每个识别到的类型有对应的必须产出
- [ ] 架构类：文件结构表完整，能直接拆 chunk
- [ ] 不碰列表覆盖上游排除项
- [ ] [人工] 用户确认设计

## 排查指南

**1. plan 阶段发现设计遗漏**
症状: plan 拆 chunk 时发现需要做设计决策
确认: plan 报告中有「需要补充设计」的标注
处理: 回退 design 补充，重走 Step 2-5

**2. 设计类型识别错误**
症状: 产出内容和实际需求不匹配
确认: 交接文件的设计类型与实际改动不符
处理: 重新识别类型，补充缺失类型的产出
