# Quality

核心价值：**全局审视** — gate 是 chunk 级局部检查，quality 从整个项目视角评估这次变更让项目变好了还是变差了。局部都通过不代表整体没问题。

> 本文件是执行指令。必须按「步骤」段逐步执行，每步验证「完成标志」后才进下一步。

<HARD-RULE>
每条结论必须附命令输出或数据作为证据。
不接受「应该没问题」「估计通过」等无证据断言。
证据先于断言，始终。
</HARD-RULE>

## 输入

| 字段 | 来源 | 必需 | 校验 |
|------|------|------|------|
| plan anchor | plan 阶段 | 是 | `{sprint}/anchors/plan.md` 存在 |
| execute 指标 | DB | 是 | sprint_chunks 有数据 |
| sprint ID | entry.md | 是 | DB 中 running + execute completed |

## 输出

| 产出 | 路径 | 消费方 | 聚焦内容 |
|------|------|--------|---------|
| 交接 | `{sprint}/handoffs/quality.md` | review/insight 消费 | 六维度结论 + 偏差 + 风险处置 + 建议 |
| 摘要 | `{sprint}/reports/quality.md` | 人确认 | 质量报告（全部证据） |

## 步骤

### Step 1: 全量测试

**做什么**: 跑项目全部测试（不是模块测试）
**怎么做**: 按 CLAUDE.md 的测试命令用 Bash 执行
**完成标志**: 贴出测试命令 + 完整输出 + 通过/失败结论

### Step 2: Anchor 全量验证

**做什么**: 重跑 invariants + boundary drift（不复用 gate 增量结果）
**怎么做**:
```bash
bash "$SPRINT_PLUGIN/scripts/sprint-ctl.sh" anchor-check "{sprint-id}"
```
**完成标志**: 贴出命令输出 + 结论

### Step 3: 风险评估

**做什么**: 对照 plan 阶段暴露的风险，逐条确认
**怎么做**: 读 `{sprint}/anchors/plan.md` 的 Risks 段，逐条检查：
- 触发信号是否出现了？
- 出现了 -> 影响评估 + 应对措施执行情况
- 没出现 -> 确认

**完成标志**: 每条风险有处置结论

### Step 4: 偏差分析

**做什么**: 对比计划 vs 实际
**怎么做**:
- diff 预算 vs 实际 diff（查 DB sprint_chunks）
- 文件范围：计划改的 vs 实际改的（design handoff vs git diff）
- chunk 数：计划 vs 实际（含跳过和重试）
- 时间预估 vs 实际

**完成标志**: 偏差表完成，偏差 > 20% 的项有解释

### Step 5: 六维度影响面评估

**做什么**: 从项目视角评估变更影响
**怎么做**:

| 维度 | 评估什么 | 怎么找证据 |
|------|---------|-----------|
| **复杂度** | 代码更复杂还是更简洁？ | `find` 统计文件数变化，`wc -l` 统计行数变化 |
| **稳定性** | 引入不稳定因素？ | 测试覆盖、error path、边界 case |
| **功能性** | 完整实现需求？ | 对照 brainstorm handoff 需求点逐条验证 |
| **性能** | 引入性能问题？ | 热路径分析、数据量级评估 |
| **质量** | 代码质量上升还是下降？ | TODO/HACK 变化（`grep -r` 计数） |
| **安全** | 引入安全风险？ | 输入验证、权限检查 |

**完成标志**: 六维度每个有结论 + 证据

### Step 6: 建议

**做什么**: 基于以上分析给出建议
**怎么做**: 建议分类：
- 可以合并：所有检查通过，无严重偏差
- 需要修复：列出具体修复项
- 建议回退：发现严重问题

**完成标志**: 有明确的「合并/修复/回退」建议

### Step 7: 写两份产出

**交接文件** `{sprint}/handoffs/quality.md`:
```markdown
# Quality -- {主题}

## 风险处置
| 风险 | 状态 | 处置 |
|------|------|------|

## 偏差
| 项 | 计划 | 实际 | 偏差 |
|----|------|------|------|

## 六维度结论
| 维度 | 结论 |
|------|------|

## 建议
{合并/修复/回退 + 理由}
```

**摘要文件** `{sprint}/reports/quality.md`:
```markdown
# Quality -- {主题}
> 状态: [ok]

## 测试
{命令 + 输出 + 结论}

## Anchor 验证
{命令 + 输出 + 结论}

## 风险
{逐条处置}

## 偏差
{表格 + 解释}

## 影响面
{六维度，每个有证据}

## 建议
{结论}
```

**完成标志**: 两份文件存在 + 每条结论有证据

### Step 8: 用户确认 [GATE:review]

**做什么**: 请用户 review 质量报告
**完成标志**: 用户确认 `[ok]`

- [ ] anchor 验证有命令输出证据

## 排查指南

**1. 全量测试失败但 gate G2 通过**
症状: quality 测试挂了但 execute 没报错
确认: 对比测试命令差异（模块 vs 全量）
处理: 记录差异，建议 execute gate 加全量测试
