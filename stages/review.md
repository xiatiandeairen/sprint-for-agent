# Review

核心价值：**知识传递** — 用户不看代码不知道改了什么、为什么这样改。如果跳过，用户只知道「任务完成了」但无法做有效的验收判断。

> 本文件是执行指令。必须按「步骤」段逐步执行，每步验证「完成标志」后才进下一步。

<HARD-RULE>
review 的价值在于用户看完后能回答「为什么这个文件要这样改」，而不只是「这个文件改了」。
如果讲解只有 diff summary 没有 why，说明没做到。
</HARD-RULE>

## 输入

| 字段 | 来源 | 必需 | 校验 |
|------|------|------|------|
| quality handoff | quality 阶段 | 有则读 | `{sprint}/handoffs/quality.md` 存在 |
| git diff | git | 是 | 有变更 |
| sprint ID | entry.md | 是 | DB 中 running |

## 输出

| 产出 | 路径 | 消费方 | 聚焦内容 |
|------|------|--------|---------|
| 交接 | `{sprint}/handoffs/review.md` | insight 消费 | 争议点 + 自审发现 |
| 摘要 | `{sprint}/reports/review.md` | 人看 | 逐文件讲解 + 评价 |

## 步骤

### Step 1: 变更概览

**做什么**: 全量 diff 统计
**怎么做**: `git diff` 统计文件数、行数、模块分布
**完成标志**: 概览输出 + 与 plan anchor 的 Goal 对照

### Step 2: 逐文件讲解

**做什么**: 每个变更文件讲「改了什么 + 为什么这样改」
**怎么做**:
- 读每个变更文件的 diff
- 讲解：改了什么 -> 为什么（引用 research/design 的决策）
- 标注争议点（有 trade-off 的实现选择）
- 遗漏检查：`git diff --name-only` 的文件数 == 讲解的文件数

**完成标志**: 所有变更文件有讲解 + 每个讲解有 why

### Step 3: AI 自审

**做什么**: 从代码质量角度审查
**怎么做**:
- 命名一致性：变量/函数命名是否与项目现有风格一致
- 错误处理：error path 是否完整
- 边界 case：空值、极端值、并发是否处理
- 模式一致性：是否遵循项目已有的设计模式

每个维度给出结论（OK / 有问题 + 具体位置）

**完成标志**: 四个维度都有结论

### Step 4: 用户评价 [BRANCH]

**做什么**: 收集用户反馈

| 条件 | 路径 |
|------|------|
| auto 模式且自审四维度全 OK | → [INFO] 输出 "[自动通过] 自审无问题"，继续 |
| 否则 | → [STOP:respond] 问用户：哪些满意？哪些需改进？有没有要修复的？ |

**完成标志**: auto: 已输出 / 用户给出反馈

### Step 5: 写两份产出

**交接文件** `{sprint}/handoffs/review.md`:
```markdown
# Review -- {主题}

## 争议点
- {trade-off 选择}

## 自审发现
| 维度 | 结论 | 详情 |
|------|------|------|

## 用户反馈
满意: {列出}
改进: {列出}
```

**摘要文件** `{sprint}/reports/review.md`:
```markdown
# Review -- {主题}
> 状态: [ok]

## 变更概览
{文件数, 行数, 模块}

## 讲解
### {file_path}
改了什么: ...
为什么: {引用 research/design 决策}

## 自审
{四维度结论}

## 用户评价
{反馈}
```

**完成标志**: 两份文件存在

## 提前终止

| 条件 | 行为 |
|------|------|
| 变更 < 10 行 | 简化：只 Step 1 + Step 4 |
| 用户说「不需要讲解」 | skip |

## 验证清单 [CHECKLIST]

- [ ] 两份文件存在 + reports 状态 `[ok]`
- [ ] 所有变更文件有讲解
- [ ] 每个讲解有 why（不只是 what）
- [ ] 自审四维度有结论
- [ ] [人工] 用户给出评价

## 排查指南

**1. 用户发现严重问题要回退**
症状: Step 4 用户要求修复
处理: 回到 execute 创建修复 chunk，之后重跑 quality + review
