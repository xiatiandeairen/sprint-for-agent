# Sprint Run — 智能路由

分析任务复杂度，自动选择流程。

## 输入

用户提供任务描述：$ARGUMENTS

## 复杂度判断

快速评估以下信号（读相关代码，不全量扫描）：

| 信号 | 小 | 中 | 大 |
|------|----|----|-----|
| 预估影响文件数 | 1-2 | 3-5 | 6+ |
| 需要架构决策 | 否 | 局部 | 跨模块 |
| 方向确定性 | 明确 | 基本明确 | 需探索 |
| 有无先例模式 | 有现成模式 | 类似但需适配 | 全新 |

### 判断规则

**小任务**（任意两项命中）：
- 影响 1-2 个文件
- 机械性替换/重命名/删除
- 项目中有完全相同的模式可复制

→ 流程：`sprint:plan` → `sprint:auto`

**中任务**（默认）：
- 影响 3-5 个文件
- 方向明确但需要设计文件结构/接口
- 不需要探索多方案

→ 流程：`sprint:design` → `sprint:plan` → `sprint:go`

**大任务**（任意一项命中）：
- 影响 6+ 个文件
- 方向不确定，有多个可行方案
- 跨 3+ 模块的架构变更
- 用户明确表示需要探索

→ 流程：`sprint:brainstorm` → `sprint:design` → `sprint:plan` → `sprint:go`

## 执行

1. 读取任务描述和相关代码
2. 输出复杂度判断：

```
📊 任务评估：<描述>
  影响范围：~N 个文件
  架构决策：是/否
  确定性：高/中/低
  判定：小/中/大 → 流程：X → Y → Z
```

3. 等用户确认：✅ 同意 / 🔄 调整流程
4. 确认后依次调用对应的 sprint 子命令（使用 Skill tool）
5. 每个阶段结束暂停确认后进入下一阶段
